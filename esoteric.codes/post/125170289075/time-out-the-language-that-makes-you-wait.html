Time Out: The language that makes you wait

2015-07-27

TimeOut: Hello World

In Time Out, each command tells the interpreter not to do anything, for a specific amount of time. Every line is in this format:

timeOut.Set(479);

This is the only valid line of code, a time out set for a certain number of steps (here 479). Meanwhile, the interpreter cycles through a list of commands, at a configurable speed (default step rate is 30 milliseconds). Think of it as a clock with a different command at each tick. When the time out is up (479 * the step rate of 30 =  a little more than 14 seconds), whatever command the interpreter is pointing at fires: it might be a command to pop from the stack, print to the screen, etc. Then it reads the next line of code asking it to wait again, and so on.

It was inspired by David Morgan-Mar’s Whenever, a language which disrupted the flow of code to much different effect (there, programmers have to construct code which might be run in any order). Time Out offsets the action of the interpreter from the present of code – but it does this with its execution environment (the Web) in mind.

Time Out runs in the browser (it’s written in JS). If other processes (or a slow browser) cause the current time out to last too long, we might miss the command we want and execute a different one, further in the cycle. In effect, it means that the programmer is forced to run the Time Out program in the active tab and not have other processes running that might interfere with it. The experience of using Time Out is one of staring at a single active tab in the browser, waiting for it to execute – if you get bored and bring up another program, it may crash. If you open another browser tab, it almost certainly will. 

Hello World attempt

The “speed” of the Time Out interpreter is configurable – 30 ms is the speed which my computer can run Time Out in Chrome without messing up. People with shorter attention spans or faster machines can configure it accordingly.
